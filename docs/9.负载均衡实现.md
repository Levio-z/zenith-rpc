# 一、需求分析

## 1.1 为什么需要负载均衡？

在分布式系统中，同一个服务通常部署在多个节点上。当消费者需要调用服务时，需要从多个服务提供者中选择一个来处理请求。

### 问题场景

```
┌─────────┐
│ 消费者   │
└────┬────┘
     │
   如何选择？
     │
     ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 服务A    │  │ 服务B    │  │ 服务C    │
│ 192.x.x.1│  │ 192.x.x.2│  │ 192.x.x.3│
└─────────┘  └─────────┘  └─────────┘
```

### 负载均衡的作用

1. **流量分发**：将请求均匀分布到多个服务节点
2. **提高吞吐**：充分利用多节点的处理能力
3. **故障转移**：某个节点故障时，自动选择其他节点
4. **扩展性**：新增节点时，自动纳入负载均衡范围

# 二、设计方案

## 2.1 负载均衡器接口设计

```java
package com.zenith.zenithrpc.loadbalancer;

import com.zenith.zenithrpc.model.ServiceMetaInfo;

import java.util.List;
import java.util.Map;

/**
 * 负载均衡器（消费端使用）
 */
public interface LoadBalancer {

    /**
     * 选择服务调用
     *
     * @param requestParams       请求参数
     * @param serviceMetaInfoList 可用的服务列表
     * @return 选中的服务节点
     */
    ServiceMetaInfo selectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList
    );

    /**
     * 默认选择实现，包含基础校验
     */
    default ServiceMetaInfo defaultSelectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList
    ) {
        if (serviceMetaInfoList.isEmpty()) {
            return null;
        }
        // 只有一个服务，无需负载均衡
        if (serviceMetaInfoList.size() == 1) {
            return serviceMetaInfoList.get(0);
        }
        return selectService(requestParams, serviceMetaInfoList);
    }
}
```

**设计要点**：

- 泛型参数支持：可以传入请求参数用于更智能的负载均衡
- 默认实现：提供基础校验逻辑，子类只需关注核心算法

# 三、常用算法实现

## 3.1 随机负载均衡器

```java
package com.zenith.zenithrpc.loadbalancer;

import com.zenith.zenithrpc.model.ServiceMetaInfo;

import java.util.List;
import java.util.Map;
import java.util.concurrent.ThreadLocalRandom;

/**
 * 随机负载均衡器
 */
public class RandomLoadBalancer implements LoadBalancer {

    @Override
    public ServiceMetaInfo selectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList
    ) {
        int index = Math.abs(ThreadLocalRandom.current().nextInt())
                     % serviceMetaInfoList.size();
        return serviceMetaInfoList.get(index);
    }
}
```

### 算法原理

```
随机选择：rand() % n

示例：3个服务节点
- 随机数 7  -> 7 % 3 = 1 -> 服务B
- 随机数 12 -> 12 % 3 = 0 -> 服务A
- 随机数 5  -> 5 % 3 = 2 -> 服务C
```

### 特点

| 优点                   | 缺点                 |
| ---------------------- | -------------------- |
| 实现简单               | 无法保证请求均匀分布 |
| 适合节点性能一致的场景 | 短时间可能不均衡     |

### 优化：加权随机

```java
// 扩展思路：根据权重随机
public ServiceMetaInfo selectService(...) {
    // 权重配置：service -> weight
    // 根据权重计算分布
}
```

## 3.2 轮询负载均衡器

```java
package com.zenith.zenithrpc.loadbalancer;

import com.zenith.zenithrpc.model.ServiceMetaInfo;

import java.util.List;
import java.util.Map;
import java.util.concurrent.atomic.AtomicInteger;

/**
 * 轮询负载均衡器
 */
public class RoundRobinLoadBalancer implements LoadBalancer {

    // 原子计数器，保证线程安全
    private final AtomicInteger count = new AtomicInteger(0);

    @Override
    public ServiceMetaInfo selectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList
    ) {
        // 取模实现轮询
        int index = count.getAndIncrement() % serviceMetaInfoList.size();
        return serviceMetaInfoList.get(index);
    }
}
```

### 算法原理

```
请求序列：1, 2, 3, 4, 5, 6, 7, 8
服务节点：A, B, C

分配结果：
1 -> A
2 -> B
3 -> C
4 -> A (循环)
5 -> B
6 -> C
7 -> A
8 -> B
```

### 特点

| 优点                   | 缺点                       |
| ---------------------- | -------------------------- |
| 实现简单               | 无法根据节点性能差异调整   |
| 请求分布均匀           | 节点变更时可能导致短期不均 |
| 适合节点性能一致的场景 |                            |

### 关键点：线程安全

使用 `AtomicInteger` 保证多线程环境下的计数器安全：

- `getAndIncrement()`：先取值后自增
- 线程安全：避免多线程竞争导致计数错误

# 四、一致性哈希

## 4.1 为什么需要一致性哈希？

### 普通哈希的问题

```java
// 普通哈希
index = hash(key) % nodes.length

// 问题场景
假设有 3 个节点：hash(key) % 3
- 节点 C 宕机后，nodes.length = 2
- 大量 key 重新映射到不同节点
- 导致缓存大量失效
```

### 一致性哈希的优势

- **最小影响**：节点变更只影响少部分请求
- **单调性**：新增节点只影响部分请求，不会导致大范围重新映射

## 4.2 算法原理

### 哈希环

```
                    0
                    |
                    |
    360 ────────────┼──────────── 0
        \           |           /
         \          |          /
          \         |         /
           \        |        /
            \       |       /
             \      |      /
              \     |     /
               \    |    /
                \   |   /
                 \  |  /
                  \ | /
                   \|/
                  180
```

### 虚拟节点

为每个物理节点创建多个虚拟节点，使分布更均匀：

```
真实节点：      虚拟节点：
  Node A    ->   A#0, A#1, A#2, ..., A#99
  Node B    ->   B#0, B#1, B#2, ..., B#99
  Node C    ->   C#0, C#1, C#2, ..., C#99
```

一致性哈希是一种分布式哈希算法，通过将哈希空间映射为环状结构，使节点和数据均分布在环上。数据按照顺时针方向映射到最近的节点，从而在节点变更时，仅影响局部数据迁移，而不是全量重新分布。

## 4.3 一致性哈希实现

```java
package com.zenith.zenithrpc.loadbalancer;

import com.zenith.zenithrpc.model.ServiceMetaInfo;

import java.util.List;
import java.util.Map;
import java.util.TreeMap;

/**
 * 一致性哈希负载均衡器
 */
public class ConsistentHashLoadBalancer implements LoadBalancer {

    /**
     * 一致性 Hash 环，存放虚拟节点
     * TreeMap 有序，便于实现 "顺时针查找"
     */
    private final TreeMap<Integer, ServiceMetaInfo> virtualNodes =
        new TreeMap<>();

    // 虚拟节点数
    private final int VIRTUAL_NODE_COUNT = 100;

    @Override
    public ServiceMetaInfo selectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList
    ) {
        // 清空旧的虚拟节点
        virtualNodes.clear();

        // 构建哈希环：添加所有虚拟节点
        for (ServiceMetaInfo serviceMetaInfo : serviceMetaInfoList) {
            for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
                // 虚拟节点 key = 真实节点地址 + 序号
                int hashCode = getHashCode(
                    serviceMetaInfo.getServiceAddress() + "#" + i
                );
                virtualNodes.put(hashCode, serviceMetaInfo);
            }
        }

        // 计算请求参数的哈希值
        int hash = getHashCode(requestParams);

        // 选择最接近且大于请求 hash 的虚拟节点
        if (virtualNodes.ceilingKey(hash) != null) {
            return virtualNodes.get(virtualNodes.ceilingKey(hash));
        }

        // 如果没有大于等于请求 hash 的节点，返回环首部（最小节点）
        return virtualNodes.firstEntry().getValue();
    }

    /**
     * 哈希函数
     */
    private int getHashCode(Object key) {
        return key.hashCode();
    }
}
```

### 算法图解

```
请求 key: "user:123"
请求 hash: 150

哈希环结构：
        0
        |
    50  |  300
        |
    100 |  250
        |
    150 |  200
        |
       哈希环

ceilingKey(150) 查找 >= 150 的最小 key
结果: 160 -> 返回对应的服务节点
```

### 特点

| 优点               | 缺点             |
| ------------------ | ---------------- |
| 节点变更影响范围小 | 实现相对复杂     |
| 支持请求参数哈希   | 节点分布可能不均 |
| 适合缓存场景       |                  |

# 五、可扩展设计：SPI + 工厂模式

## 5.1 负载均衡器常量

```java
package com.zenith.zenithrpc.loadbalancer;

/**
 * 负载均衡器常量
 */
public interface LoadBalancerKeys {
    String ROUND_ROBIN = "roundRobin";
    String RANDOM = "random";
    String CONSISTENT_HASH = "consistentHash";
}
```

## 5.2 工厂模式

```java
package com.zenith.zenithrpc.loadbalancer;

import com.zenith.zenithrpc.serializer.SpiLoader;

/**
 * 负载均衡器工厂
 */
public class LoadBalancerFactory {

    static {
        // 启动时加载所有负载均衡器
        SpiLoader.load(LoadBalancer.class);
    }

    /**
     * 默认负载均衡器
     */
    private static final LoadBalancer DEFAULT_LOAD_BALANCER =
        new RoundRobinLoadBalancer();

    /**
     * 获取负载均衡器实例
     *
     * @param loadBalancerKey 负载均衡器 key
     * @return 负载均衡器实例
     */
    public static LoadBalancer getLoadBalancer(String loadBalancerKey) {
        return SpiLoader.getInstance(LoadBalancer.class, loadBalancerKey);
    }
}
```

## 5.3 SPI 配置文件

**META-INF/rpc/system/com.zenith.zenithrpc.loadbalancer.LoadBalancer**

```
roundRobin=com.zenith.zenithrpc.loadbalancer.RoundRobinLoadBalancer
random=com.zenith.zenithrpc.loadbalancer.RandomLoadBalancer
consistentHash=com.zenith.zenithrpc.loadbalancer.ConsistentHashLoadBalancer
```

## 5.4 使用示例

```java
// 根据配置获取负载均衡器
String loadBalancerKey = rpcConfig.getLoadBalancer();
LoadBalancer loadBalancer = LoadBalancerFactory.getLoadBalancer(loadBalancerKey);

// 调用负载均衡
ServiceMetaInfo selectedService = loadBalancer.defaultSelectService(
    requestParams,
    serviceList
);
```

# 六、算法对比

| 算法       | 时间复杂度 | 适用场景         | 特点           |
| ---------- | ---------- | ---------------- | -------------- |
| 随机       | O(1)       | 节点性能一致     | 简单，可能不均 |
| 轮询       | O(1)       | 节点性能一致     | 均匀，无状态   |
| 一致性哈希 | O(log n)   | 缓存、分布式存储 | 最小化迁移     |

# 七、扩展思路

### 7.1 加权负载均衡

根据节点性能分配不同权重：

```java
public interface LoadBalancer {
    // 支持权重参数
    ServiceMetaInfo selectService(
        Map<String, Object> requestParams,
        List<ServiceMetaInfo> serviceMetaInfoList,
        Map<String, Integer> weights  // 新增
    );
}
```

### 7.2 最少连接数

将请求发送到当前连接数最少的节点：

```java
public class LeastConnectionsLoadBalancer implements LoadBalancer {
    // 跟踪每个节点的活跃连接数
    private Map<String, AtomicInteger> connectionCounts;
}
```

### 7.3 本地优先

优先选择本机或同机房的节点，减少网络延迟：

```java
public class LocalFirstLoadBalancer implements LoadBalancer {
    // 检测节点是否在本地或同机房
    private boolean isLocal(ServiceMetaInfo service);
}
```

### 7.4 健康检查

集成健康检查，过滤不健康的节点：

```java
public interface HealthChecker {
    boolean isHealthy(ServiceMetaInfo service);
}
```

# 八、总结

## 8.1 实现要点

1. **接口设计**：抽象出 `LoadBalancer` 接口，便于扩展
2. **SPI 机制**：支持动态加载自定义负载均衡器
3. **工厂模式**：统一管理负载均衡器实例
4. **线程安全**：使用 `AtomicInteger` 等保证并发安全

## 8.2 配置使用

```yaml
rpc:
  loadbalancer: roundRobin # 可选: roundRobin, random, consistentHash
```

## 8.3 选择建议

| 场景         | 推荐算法    |
| ------------ | ----------- |
| 节点性能一致 | 轮询        |
| 无状态服务   | 随机        |
| 缓存/存储    | 一致性哈希  |
| 微服务调用   | 轮询 + 重试 |
