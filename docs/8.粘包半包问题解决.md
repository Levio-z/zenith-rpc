# 一、问题背景

## 1.1 什么是 TCP 粘包半包？

TCP（Transmission Control Protocol）是一种面向连接的、可靠的、基于字节流的传输层通信协议。由于 TCP 协议的自身特性，在网络编程中经常遇到"粘包"和"半包"问题。

### 粘包问题

**定义**：多个独立的消息被合并成一个 TCP 数据包发送。

**场景示例**：

```
客户端发送了：
  - 消息A: "Hello"
  - 消息B: "World"

服务端可能收到：
  - "HelloWorld"  (粘在一起)
```

### 半包问题

**定义**：一个消息被拆分成多个 TCP 数据包发送。

**场景示例**：

```
客户端发送了：
  - 消息A: "HelloWorld" (较长的消息)

服务端可能分多次收到：
  - 第一包: "Hel"
  - 第二包: "loWorld"
```

## 1.2 为什么会出现粘包半包？

TCP 是一个字节流协议，没有消息边界。它会：

- **Nagle 算法**：将多个小数据包合并成一个发送，提高网络效率
- **缓冲区**：根据网络状况和窗口大小自动分包
- **应用层不感知**：应用层看到的只是一串连续字节流

## 1.3 粘包半包的危害

- 消息解析错误：无法正确识别消息边界
- 数据丢失：半包可能导致数据丢失
- 业务逻辑错误：消息错乱

# 二、解决方案

## 2.1 常见解决方案

### 方案一：固定长度

每个消息都是固定长度。

```
优点：解析简单
缺点：浪费带宽，限制消息长度
```

### 方案二：分隔符

使用特殊分隔符（如 `\n`）标识消息结束。

```
优点：实现简单
缺点：需要处理转义，分隔符不能出现在消息内容中
```

### 方案三：Length Field（长度字段）【本框架采用】

在消息头部包含消息体长度。

```
+------------+---------------+
|   Header   |     Body      |
| 4 bytes    |   N bytes     |
+------------+---------------+
| bodyLength |    content     |
+------------+---------------+
```

```
优点：高效、灵活、无消息内容限制
```

# 三、Vert.x 解决方案

## 3.1 RecordParser

Vert.x 提供了 `RecordParser` 工具，专门用于处理粘包半包问题。它支持：

- 固定长度模式
- 可变长度模式
- 混合模式

## 3.2 装饰者模式实现

我们使用**装饰者模式**封装 `RecordParser`，提供统一的 `Handler<Buffer>` 接口。

```java
package com.zenith.zenithrpc.server.tcp;

import com.zenith.zenithrpc.protocol.ProtocolConstant;
import io.vertx.core.Handler;
import io.vertx.core.buffer.Buffer;
import io.vertx.core.parsetools.RecordParser;

/**
 * TCP 缓冲区处理包装器（装饰者模式）
 * 用于解决粘包半包问题
 */
public class TcpBufferHandlerWrapper implements Handler<Buffer> {

    private final RecordParser recordParser;

    public TcpBufferHandlerWrapper(Handler<Buffer> bufferHandler) {
        recordParser = initRecordParser(bufferHandler);
    }

    @Override
    public void handle(Buffer buffer) {
        recordParser.handle(buffer);
    }

    /**
     * 初始化 RecordParser
     * 采用混合模式：先读固定头部，再读变长消息体
     */
    private RecordParser initRecordParser(Handler<Buffer> bufferHandler) {
        // 构造 parser，初始模式为读取固定长度的消息头
        RecordParser parser = RecordParser.newFixed(ProtocolConstant.MESSAGE_HEADER_LENGTH);

        parser.setOutput(new Handler<Buffer>() {
            // 当前要读取的消息体长度
            int size = -1;
            // 存放完整的消息（头 + 体）
            Buffer resultBuffer = Buffer.buffer();

            @Override
            public void handle(Buffer buffer) {
                if (-1 == size) {
                    // 第一阶段：读取消息头（固定17字节）
                    // 从头部第13字节读取4字节获取消息体长度
                    size = buffer.getInt(13);

                    // 切换为固定长度模式，读取 bodyLength 字节的数据
                    parser.fixedSizeMode(size);

                    // 将头部信息写入结果缓冲区
                    resultBuffer.appendBuffer(buffer);
                } else {
                    // 第二阶段：读取消息体
                    // 将消息体写入结果缓冲区
                    resultBuffer.appendBuffer(buffer);

                    // 已拼接为完整消息，交给业务处理器
                    bufferHandler.handle(resultBuffer);

                    // 重置：切换回读取头部的模式
                    parser.fixedSizeMode(ProtocolConstant.MESSAGE_HEADER_LENGTH);
                    size = -1;
                    resultBuffer = Buffer.buffer();
                }
            }
        });

        return parser;
    }
}
```

# 四、工作流程详解

## 4.1 协议头结构回顾

```
Position    Length    Field         Description
--------    ------    -----         -----------
0           1         magic         魔数
1           1         version       版本号
2           1         serializer    序列化器
3           1         type          消息类型
4           1         status        状态
5           8         requestId     请求ID
13          4         bodyLength    消息体长度
17          N         body          消息体
```

## 4.2 解析流程

```
收到数据: | HEADER(17) | BODY(N) |
                ↓
         RecordParser 先读取 HEADER
                ↓
         从 HEADER[13-16] 解析 bodyLength = N
                ↓
         RecordParser 切换为 fixedSizeMode(N)
                ↓
         读取 N 字节的 BODY
                ↓
         拼接为完整消息，触发回调
                ↓
         重置状态，继续接收下一条消息
```

## 4.3 时序图

```
客户端                                    服务端
  |                                        |
  | ---- TCP 数据流 --------------------> |
  |   [粘包: MSG1][MSG2][半包 MSG3 PART1] |
  |                                        |
  |   RecordParser 开始解析                |
  |   - 读取 17 字节 HEADER                |
  |   - 解析 bodyLength = 100             |
  |   - 切换 fixedSizeMode(100)           |
  |                                        |
  | <-------- MSG1 完整数据 ----------- |
  |      (触发业务处理回调)                 |
  |                                        |
  |   RecordParser 继续解析                |
  |   - 读取 17 字节 HEADER                |
  |   - 解析 bodyLength = 80              |
  |   - 读取部分 BODY                      |
  |                                        |
  | ---- 下一包数据 ---------------------> |
  |   [半包 MSG3 PART2]                   |
  |                                        |
  |   继续完成 MSG3 的读取                 |
  |                                        |
  | <-------- MSG2 完整数据 ----------- |
  |                                        |
  | <-------- MSG3 完整数据 ----------- |
  |                                        |
```

# 五、装饰者模式应用

## 5.1 模式说明

**装饰者模式（Decorator Pattern）**：动态地给对象添加一些额外的职责。

**为什么使用装饰者模式？**

- `RecordParser` 本身实现了 `Handler<Buffer>` 接口
- 但它的功能是分片解析，不是直接处理业务
- 需要包装一层，将解析后的完整消息交给业务处理器

## 5.2 类图

```
                <<interface>>
                Handler<Buffer>
                      │
                      │
        ┌─────────────┴─────────────┐
        │                           │
        ▼                           ▼
TcpBufferHandlerWrapper      RecordParser
(装饰者)                     (被装饰者)
        │                           │
        │ 持有                      │
        └───────────┬───────────────┘
                    │ 组合
                    ▼
              处理粘包半包
              返回完整消息
```

## 5.3 优势

1. **单一职责**：`TcpBufferHandlerWrapper` 专注于粘包半包处理
2. **可复用**：业务处理器无需关心底层解析细节
3. **灵活组合**：可以装饰不同的 Handler

# 六、客户端应用

客户端同样需要处理粘包半包问题：

```java
// 接收响应
TcpBufferHandlerWrapper bufferHandlerWrapper =
    new TcpBufferHandlerWrapper(buffer -> {
        try {
            ProtocolMessage<RpcResponse> rpcResponseProtocolMessage =
                (ProtocolMessage<RpcResponse>)
                    ProtocolMessageDecoder.decode(buffer);
            responseFuture.complete(rpcResponseProtocolMessage.getBody());
        } catch (IOException e) {
            throw new RuntimeException("协议消息解码错误");
        }
    });
socket.handler(bufferHandlerWrapper);
```

**关键点**：

- 客户端和服务端使用相同的粘包半包处理逻辑
- `TcpBufferHandlerWrapper` 保证了消息的完整性

# 八、总结

## 8.1 关键设计点

1. **协议头固定长度**：17字节，便于快速定位和解析
2. **Length Field 方案**：通过 `bodyLength` 字段标识消息体长度
3. **RecordParser**：Vert.x 提供的工具，简化粘包半包处理
4. **装饰者模式**：灵活包装 Handler，统一接口

## 8.2 解决的问题

| 问题     | 解决方案                      |
| -------- | ----------------------------- |
| 粘包     | RecordParser 固定长度模式解析 |
| 半包     | 分阶段读取：先读头，再读体    |
| 完整消息 | 拼接后触发业务回调            |
