# 一、需求分析

## 1.1 为什么需要容错机制？

分布式系统中，服务之间的调用可能出现各种异常：网络超时、服务宕机、服务不可用等。容错机制决定了在故障发生时系统如何响应。

### 问题场景

```
┌─────────┐                      ┌─────────┐
│ 消费者   │  ──请求──>           │ 提供者A   │
└─────────┘                      └─────────┘
     │                              │
     │                              │ X 服务A宕机
     │                              │
     ? 如何处理？
```

### 容错的价值

1. **系统韧性**：系统在部分组件故障时仍能运行
2. **用户体验**：降级处理，避免直接报错
3. **可维护性**：故障隔离，防止问题扩散

## 1.2 容错 vs 重试

| 维度 | 重试机制 | 容错机制 |
|------|---------|---------|
| 时机 | 调用失败时 | 调用失败后 |
| 目标 | 恢复调用 | 处理失败结果 |
| 策略 | 再次尝试 | 返回默认值/备用方案 |
| 适用 | 瞬时故障 | 持久故障 |

**执行顺序**：
```
调用失败
    │
    ▼
┌─────────────────┐
│ 重试机制        │ ← 重试多次仍失败
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 容错机制        │ ← 处理最终失败
└─────────────────┘
```

# 二、设计方案

## 2.1 容错策略接口

```java
package com.zenith.zenithrpc.fault.tolerant;

import com.zenith.zenithrpc.model.RpcResponse;

import java.util.Map;

/**
 * 容错策略接口
 */
public interface TolerantStrategy {

    /**
     * 处理容错响应
     *
     * @param context 上下文信息
     * @param e        异常信息
     * @return 容错后的响应
     */
    RpcResponse doResponse(Map<String, Object> context, Exception e);
}
```

**设计说明**：
- 接收异常信息，用于分析错误类型
- 接收上下文信息，用于更智能的容错处理
- 返回 `RpcResponse`，保证调用链正常返回

# 三、容错策略实现

## 3.1 快速失败策略（Fail Fast）

立即抛出异常，不做任何处理。最简单的容错策略。

```java
package com.zenith.zenithrpc.fault.tolerant;

import com.zenith.zenithrpc.model.RpcResponse;

import java.util.Map;

/**
 * 快速失败策略
 * 直接抛出异常，不做任何容错处理
 */
public class FailFastTolerantStrategy implements TolerantStrategy {

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        // 直接抛出运行时异常
        throw new RuntimeException("服务报错", e);
    }
}
```

### 特点

| 优点 | 缺点 |
|------|------|
| 实现简单 | 调用方直接感知错误 |
| 响应速度快 | 用户体验差 |
| 便于问题定位 | 可能导致级联失败 |

### 适用场景

- 核心业务流程
- 对数据一致性要求高的操作
- 开发调试阶段

### 时序图

```
消费者 ──> 提供者 ──> [异常]
            │
            ▼
         快速失败
            │
            ▼
         抛出异常 ──> 消费者感知
```

## 3.2 安全失败策略（Fail Safe）

静默处理异常，返回一个安全的默认值，不影响主流程。

```java
package com.zenith.zenithrpc.fault.tolerant;

import com.zenith.zenithrpc.model.RpcResponse;
import lombok.extern.slf4j.Slf4j;

import java.util.Map;

/**
 * 安全失败策略
 * 静默处理异常，返回空响应
 */
@Slf4j
public class FailSafeTolerantStrategy implements TolerantStrategy {

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        // 记录异常日志
        log.info("静默处理异常", e);
        // 返回空响应，不影响主流程
        return new RpcResponse();
    }
}
```

### 特点

| 优点 | 缺点 |
|------|------|
| 不影响主流程 | 可能丢失数据 |
| 系统可用性高 | 问题可能被隐藏 |
| 日志可追溯 | 不适合核心业务 |

### 适用场景

- 非核心功能调用
- 日志记录、统计等辅助功能
- 审计、监控等可容忍失败的场景

### 时序图

```
消费者 ──> 提供者 ──> [异常]
            │
            ▼
         安全失败
            │
            ├──> 记录日志
            │
            ▼
         返回空响应 ──> 消费者继续执行
```

# 四、容错策略对比

| 策略 | 行为 | 返回值 | 异常处理 |
|------|------|--------|---------|
| 快速失败 | 抛出异常 | 无 | 上层感知 |
| 安全失败 | 静默处理 | 空响应 | 日志记录 |

| 熔断返回 | 返回降级数据 | 降级数据 | 记录指标 |

# 五、其他容错策略

## 5.1 故障转移策略（Fail Over）

当主服务失败时，自动切换到备用服务。

```java
public class FailOverTolerantStrategy implements TolerantStrategy {

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        // 获取备用服务列表
        List<ServiceMetaInfo> backupServices = getBackupServices(context);

        for (ServiceMetaInfo backup : backupServices) {
            try {
                // 尝试调用备用服务
                return callService(backup, context);
            } catch (Exception ex) {
                // 继续尝试下一个
                continue;
            }
        }

        // 所有备用服务都失败
        throw new RuntimeException("所有服务不可用", e);
    }
}
```

## 5.2 返回缓存策略

当服务不可用时，返回历史缓存数据。

```java
public class CacheTolerantStrategy implements TolerantStrategy {

    private Map<String, RpcResponse> cache;

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        String cacheKey = generateCacheKey(context);

        // 尝试从缓存获取
        RpcResponse cached = cache.get(cacheKey);
        if (cached != null) {
            log.info("从缓存返回数据");
            return cached;
        }

        // 无缓存，返回空
        return new RpcResponse();
    }
}
```

## 5.3 降级策略

返回简化版本的响应。

```java
public class DegradeTolerantStrategy implements TolerantStrategy {

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        RpcResponse response = new RpcResponse();
        response.setMessage("服务降级，返回降级数据");

        // 返回降级数据
        Map<String, Object> degradeData = new HashMap<>();
        degradeData.put("status", "degraded");
        degradeData.put("message", "服务暂时不可用");
        response.setData(degradeData);

        return response;
    }
}
```

# 六、可扩展设计：SPI + 工厂模式

## 6.1 容错策略常量

```java
package com.zenith.zenithrpc.fault.tolerant;

/**
 * 容错策略常量
 */
public interface TolerantStrategyKeys {
    /**
     * 快速失败
     */
    String FAIL_FAST = "failFast";

    /**
     * 安全失败
     */
    String FAIL_SAFE = "failSafe";
}
```

## 6.2 容错策略工厂

```java
package com.zenith.zenithrpc.fault.tolerant;

import com.zenith.zenithrpc.serializer.SpiLoader;

/**
 * 容错策略工厂
 */
public class TolerantStrategyFactory {

    static {
        // 启动时加载所有容错策略
        SpiLoader.load(TolerantStrategy.class);
    }

    /**
     * 获取容错策略实例
     *
     * @param key 容错策略 key
     * @return 容错策略实例
     */
    public static TolerantStrategy getInstance(String key) {
        return SpiLoader.getInstance(TolerantStrategy.class, key);
    }
}
```

## 6.3 SPI 配置

**META-INF/rpc/system/com.zenith.zenithrpc.fault.tolerant.TolerantStrategy**

```
failFast=com.zenith.zenithrpc.fault.tolerant.FailFastTolerantStrategy
failSafe=com.zenith.zenithrpc.fault.tolerant.FailSafeTolerantStrategy
```

## 6.4 使用示例

```java
// 获取容错策略
String tolerantKey = rpcConfig.getTolerantStrategy();
TolerantStrategy tolerantStrategy = TolerantStrategyFactory.getInstance(tolerantKey);

// 构建上下文
Map<String, Object> context = new HashMap<>();
context.put("request", rpcRequest);
context.put("services", serviceList);

// 执行调用，失败时走容错
RpcResponse response;
try {
    response = vertxTcpClient.doRequest(rpcRequest, serviceMetaInfo);
} catch (Exception e) {
    // 调用失败，使用容错策略处理
    response = tolerantStrategy.doResponse(context, e);
}
```

# 七、重试 + 容错组合

重试和容错机制通常配合使用：

```java
public class RPCClient {

    private RetryStrategy retryStrategy;
    private TolerantStrategy tolerantStrategy;

    public RpcResponse call(RpcRequest request, ServiceMetaInfo service) {
        try {
            // 使用重试策略执行调用
            return retryStrategy.doRetry(() -> {
                return doRequest(request, service);
            });
        } catch (Exception e) {
            // 重试全部失败，使用容错策略
            Map<String, Object> context = new HashMap<>();
            context.put("request", request);
            return tolerantStrategy.doResponse(context, e);
        }
    }
}
```

### 执行流程

```
┌─────────────────────────────────────────────────────┐
│                    RPC 调用                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  第一次调用                                          │
│      │                                              │
│      ├──> 成功 ──> 返回响应                         │
│      │                                              │
│      └──> 失败                                      │
│              │                                      │
│              ▼                                      │
│         重试 (第2次)                                 │
│              │                                      │
│              ├──> 成功 ──> 返回响应                 │
│              │                                      │
│              └──> 失败                              │
```

# 八、扩展思路

## 8.1 故障转移策略

```java
public class FailOverTolerantStrategy implements TolerantStrategy {

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        List<ServiceMetaInfo> backupServices =
            (List<ServiceMetaInfo>) context.get("backupServices");

        for (ServiceMetaInfo backup : backupServices) {
            try {
                return callService(backup, context);
            } catch (Exception ignored) {
                // 尝试下一个服务
            }
        }

        throw new RuntimeException("无可用服务", e);
    }
}
```

## 8.2 限流策略

防止过载导致的服务雪崩：

```java
public class RateLimitStrategy implements TolerantStrategy {

    private RateLimiter rateLimiter;

    @Override
    public RpcResponse doResponse(Map<String, Object> context, Exception e) {
        if (!rateLimiter.tryAcquire()) {
            // 限流，返回降级响应
            return getDegradeResponse("请求被限流");
        }
        throw new RuntimeException(e);
    }
}
```

## 8.3 舱壁隔离策略

隔离不同类型的调用，防止相互影响：

```java
public class BulkheadStrategy implements TolerantStrategy {

    // 每个服务一个线程池
    private Map<String, ExecutorService> pools = new ConcurrentHashMap<>();

    public RpcResponse call(String serviceName, Callable<RpcResponse> task) {
        ExecutorService pool = pools.computeIfAbsent(
            serviceName,
            k -> Executors.newFixedThreadPool(10)
        );

        try {
            return pool.submit(task).get(timeout, TimeUnit.SECONDS);
        } catch (Exception e) {
            // 降级处理
        }
    }
}
```

# 九、最佳实践

## 9.1 策略选择原则

| 业务场景 | 推荐策略 |
|---------|---------|
| 核心支付流程 | 快速失败 |
| 查询用户信息 | 快速失败 |
| 记录日志 | 安全失败 |
| 发送消息 | 安全失败 |
| 多服务调用 | 故障转移 |
| 读多写少 | 返回缓存 |

## 9.2 配置建议

```yaml
rpc:
  # 核心业务：严格模式
  core:
    retry: fixedInterval
    retry-times: 3
    tolerant: failFast

  # 非核心业务：宽松模式
  auxiliary:
    retry: no
    tolerant: failSafe
```

## 9.3 监控指标

| 指标 | 说明 |
|------|------|
| `tolerant.fail.count` | 容错处理次数 |
| `tolerant.fail.type` | 容错类型分布 |
| `tolerant.fail.service` | 按服务分类 |
| `tolerant.failover.count` | 故障转移次数 |

# 十、总结

## 10.1 实现要点

1. **接口设计**：抽象 `TolerantStrategy` 接口，支持多种策略
2. **SPI 机制**：支持动态加载自定义容错策略
3. **工厂模式**：统一管理策略实例
4. **日志记录**：便于问题排查和监控

## 10.2 重试与容错关系

```
调用失败
    │
    ▼
┌──────────┐
│ 重试机制  │
│ (短时恢复) │
└────┬─────┘
     │
     ▼
┌──────────┐
│ 容错机制  │
│ (最终兜底) │
└──────────┘
```

## 10.3 选择建议

- **快速失败**：核心业务、数据一致性要求高
- **安全失败**：辅助功能、可用性要求高
- **故障转移**：有多个服务实例的场景
- **返回缓存**：数据变化不敏感的场景
